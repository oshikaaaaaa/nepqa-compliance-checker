ntordust — AI Engineer Assessment: Task 1 (China → Nepal)

## Overview

This project is a submission for the **Cantordust AI Engineer Assessment, Task 1**. The task involves preparing a draft Nepal import compliance file for **SunBridge Trading**, a Kathmandu-based solar equipment importer bringing grid-tied inverters in from China.

The pipeline ingests two Chinese manufacturer PDFs and the NEPQA 2025 Nepal import guideline, extracts structured data from each using an LLM agent, and then generates a professional compliance draft document suitable for review by a Nepal import agent.

---

## The Business Problem

SunBridge Trading received two manufacturer PDFs from a Chinese supplier. The PDFs may describe slightly different product variants. A local Nepal agent needs a compliance review file — structured around what Nepal import review typically examines (using NEPQA 2025 as reference) — before the shipment can proceed.

This pipeline automates the extraction and cross-referencing work, then drafts the document a human reviewer would otherwise produce manually.

## Pipeline

The pipeline runs in four sequential steps:

### Step 1 — PDF to Markdown (`step1_pdf_to_md.py`)

Extracts text and tables from PDF pages using **pdfplumber**, with a custom spanning-cell detection algorithm for tables. Output is cached to `data/extracted_md/` so re-runs skip extraction.



### Step 2a — NEPQA Requirements to JSON (`step2_nepqa_to_json.py`)

A separate LLM agent reads the NEPQA 2025 guideline pages and extracts all import requirements as a flat key-value JSON (e.g. `"thd_current": "< 5% at rated power (IEC 61727:2004)"`).

This agent uses **Tavily web search** to look up numeric values for any requirements that reference external standards (IEC, NEA grid code, etc.) without giving specific numbers. Output is cached to `data/extracted_json/nepalqa.json`.


### Step 2b — Manufacturer Specs to JSON (`step2_manufacturer_to_json.py`)

An LLM agent (GPT-4.1-mini via LangChain) reads the extracted Markdown and produces a structured JSON with:
- `models` — list of product model names found
- `general_fields` — specs shared across all models (e.g. certifications, topology)
- `models_specs` — per-model technical parameters

The agent uses the NEPQA requirements JSON as a key schema so extracted fields map directly to what Nepal review asks for. Output is cached to `data/extracted_json/`.

### Step 3 — Compliance Report Generation (`step3_generate_report.py`)

Takes all three JSON files (NEPQA requirements, Manufacturer 1, Manufacturer 2) and calls the LLM with a detailed structured prompt to produce a professional Markdown compliance draft. The report contains:

1. Cover note (memo from SunBridge to their Nepal agent)
2. Product and manufacturer overview
3. Technical specifications review table (SUN series vs CE series vs NEPQA requirements)
4. Certifications and test reports
5. Labeling review
6. Document checklist
7. Open items and discrepancies
8. Preparer's approach and limitations

## Setup

### Requirements

- Python 3.10+
- An OpenAI API key (GPT-4.1-mini)
- A Tavily API key (for web search in the NEPQA extraction step)

### Install dependencies

```bash
cd cantordust
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt # or install from venv/lib if using the bundled env
```

### Configure environment

Create a `.env` file in the project root:

```env
OPENAI_API_KEY=sk-...
TAVILY_API_KEY=tvly-...
```

### Run

```bash
python run.py
```

The pipeline uses file caching at each step — if `data/extracted_md/`, `data/extracted_json/`, and `output/compliance_report.md` already exist, those steps are skipped. Delete the relevant cached file to force a re-run of any step.

## Key Design Decisions

**Caching at every step.** PDF extraction and LLM calls are expensive. All intermediate outputs are saved to disk so partial re-runs are cheap and the pipeline is resumable.

**Separate agents for manufacturer vs. NEPQA.** The two extraction tasks have different shapes: manufacturer docs need model-level structured extraction; NEPQA needs requirement-level flat extraction with external standard lookup. Splitting them allows different prompts, tools, and output schemas.

**Web search for standards resolution.** NEPQA references IEC and NEA standards by name without always stating the numeric threshold. The NEPQA agent uses Tavily to look up the actual values (e.g. THD limits, anti-islanding trip times) so the requirements JSON contains actionable numbers, not just standard citations.

**Honest discrepancy reporting.** The report generation prompt explicitly instructs the LLM to flag CONFLICT, NOT PROVIDED, and PARTIAL assessments rather than inferring or smoothing over gaps. The two manufacturer PDFs describe different product series (3-phase SUN series vs 1-phase CE series) and this is surfaced clearly in the output.
